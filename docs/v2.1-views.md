## 1. Component library: NONE — semantic HTML required

No external UI library. No Mantine / MUI / Chakra / Radix / shadcn. Hand-built components in `app/src/components/`, scoped CSS via CSS modules (`*.module.css`).

**Semantic HTML is non-negotiable** — the "no library" path has known accessibility regression risk. Pinned patterns:

| Need | Use | NOT |
|---|---|---|
| Collapsible row | `<details><summary>` | `<div onClick toggle>` |
| Dropdown | `<select><option>` | `<div role="combobox">` |
| Modal / confirm dialog | `<dialog>` (`showModal()` / `close()`) | `<div role="dialog">` |
| SSE event feed | `<ul aria-live="polite">` | `<div>` with manual focus mgmt |
| Active nav link | `<a aria-current="page">` | `<a className="active">` (alone) |
| Status pill | `<span role="status">` for live regions | bare `<span>` |
| Form fieldset grouping | `<fieldset><legend>` | `<div>` |
| Long-running action | `<button>` + `aria-busy="true"` while in-flight | non-button clickable |

Mike has reduced vision — keyboard nav + screen reader support is real, not theoretical. Without these patterns, agents reach for div+role and accessibility regresses silently. With them, the bundle stays lean *and* keyboard nav works.

**Re-audit grep policy:** `app/src/` for `role="button"`, `role="dialog"`, `role="combobox"`, `<div onClick`, `onClick={() => …}` on non-button elements without `tabIndex`. Each hit is a re-audit blocker unless justified inline with a comment.

**Bundle target:** stay under **400KB / 120KB gzipped** post-Wave-5 (Wave 4 baseline 220KB / 69KB; Wave 5 budget +180KB / +50KB gzipped).

**Theme:** dark, matches existing Sovereignty branding (favicons + OG card from v2.0.0 Stage D + landing page). CSS custom properties defined once in `app/src/styles/theme.css`:


```
│    │  …                                                         │
│    │  FINAL    ✓ anchored    txid ABC1…F0   [explorer ↗]       │
│    │                                                            │
│    │  [Verify all rounds]                                      │
│                                                                 │
│  ▶ s71  town_hall_v1  in progress  3 rounds  ⊘ 3 pending        │
│  ▶ s99  campfire_v1   complete   12 rounds   ✗ 1 missing        │
```

### Behavior

- Top-level: collapsed list of games, sorted by `started_iso` descending (latest first).
- Each row: `<details><summary>` with game-id, ruleset, status, round count, anchor summary.
- Click to expand: rounds table with anchor status (anchored / pending / missing), txid (truncated to 4 chars + `…` + 2 chars), explorer link via `transport.explorer_tx_url(txid)` (daemon's `/games/{id}/anchor-status/{round}` returns the URL — frontend doesn't construct it).
- Anchor status icon: `✓` (anchored), `⊘` (pending), `✗` (missing). `aria-label` per cell.
- Empty state: "No games yet. Start one with `sov play campfire_v1`." (use `<EmptyState>` shared component).

### Verify-all-rounds UX

The "Verify all rounds" button kicks off a sequential per-round verification. **Sequential, not concurrent** — XRPL has implicit per-IP rate limits; parallel hits get throttled.

Per round, run two checks in series:
1. `verify_proof_local(proof_path)` — local envelope_hash recompute. Fast (CPU-only, <10ms).
2. `is_anchored_on_chain(txid, expected_hash)` — chain lookup via daemon `/games/{id}/anchor-status/{round}`. Slow (~500ms-2s).

State transitions per round:

```
idle  ─►  ⋯ verifying  ─►  ✓ verified
                       └►  ✗ failed (envelope mismatch | not on chain | unreachable)
```

Render as a column next to the existing anchor-status column. Once verification runs, the column persists for the session.

UX requirements:
- **Per-round progress indicator** — `⋯` while in-flight, `✓` or `✗` on completion. Update live as each round resolves.
- **Cancel button** appears next to "Verify all rounds" while in-flight; clicking cancels remaining rounds (already-completed results retained).
- **Cancel-on-navigate-away** — if user navigates to another route mid-verify, abort the in-flight request and clear the in-flight state. Restartable from `/audit` re-mount.
- **Session cache** — completed verification results stored in component state (or context) for the session. Re-running is the user's call ("Verify all rounds" button stays available; result cells become re-runnable on click).
- `aria-busy="true"` on the verify button while in-flight; `aria-live="polite"` region reports progress to screen readers ("Verifying round 7 of 15…").

### Daemon endpoints consumed

| Endpoint | Purpose |
|---|---|
| `GET /games` | Top-level games list |
| `GET /games/{id}` | Game state for header summary (round count, status) |
| `GET /games/{id}/proofs` | Round list for expanded view |
| `GET /games/{id}/anchor-status/{round}` | Per-round 3-state + txid + explorer_url |
| `GET /games/{id}/proofs/{round}` | Proof contents for `verify_proof_local` recompute |

### SSE consumption

| Event | Behavior |
|---|---|
| `anchor.batch_complete` | Update affected rounds: `pending` → `anchored`, store txid + explorer_url from payload. **No re-fetch.** Animate the status flip subtly. |
| `anchor.pending_added` | Update `pending_anchors` count in game summary row from payload. **No re-fetch.** |
| `game.state_changed` | If the changed game is currently expanded, fire one `GET /games/{id}` re-fetch + one `GET /games/{id}/proofs` re-fetch. Other games stay cached. |

### Round timeline

Renders as a horizontal row of dots. State per dot:
- `●` filled (`--sov-accent`) — round complete.
- `◐` half (`--sov-accent` outlined) — current active round.
- `○` empty — pending round.

Rounds 1..N where N = `state.config.max_rounds` (default 15). If max_rounds ≠ 15, scale dot count.

### Recent events log

`<ul aria-live="polite">` of last 20 SSE events, oldest dropped on overflow.

Format per row: `• {HH:MM:SS}  {event.type}  {summary}`

Summary derived per event type:

| Event | Summary |
|---|---|
| `daemon.ready` | `network: {network}, ipc_version: {ipc_version}` |
| `daemon.shutdown` | `reason: {reason}` |
| `anchor.pending_added` | `round {round}` |
| `anchor.batch_complete` | `rounds {first}–{last} → {txid_short}` |
| `game.state_changed` | `round {current_round} update` (after re-fetch resolves) |

## 4. `/settings` — network switcher with guardrails

### Layout

- **Daemon config section** (existing from Wave 4): URL, port, network, readonly, started_iso, uptime. Read-only.
- **Network switcher**: `<select>` with options `testnet` / `mainnet` / `devnet`. Below: `<button>Apply (restarts daemon)</button>`.
- **Mode switcher**: shows current readonly state. Button: "Switch to readonly" or "Switch to full mode" depending on current state. Same restart flow.
- **Daemon log location** hint (read-only text — points at platform-appropriate log path).

### Three guardrails (network switcher)

Pinned guardrails. Each must trip BEFORE the daemon-restart flow runs:

| Pre-condition | Behavior |
|---|---|
| `started_by_shell == false` (externally-started daemon) | Refuse switch. Show inline message: "Daemon was started externally — stop it via `sov daemon stop` and restart from the shell to manage networks here." Disable Apply button. |
| Active game has non-empty `pending-anchors.json` | Refuse switch. Show inline message: "Pending anchors target the current network. Run `sov anchor` to flush them first, then switch networks." Disable Apply button. |
| Switching to/from `mainnet` (either direction) | Open `<dialog>` confirm. Copy: "Mainnet anchors cost real XRP. Confirm switch from {current} → {target}?" Apply only on user confirm. |

The `started_by_shell` value is read via `daemon_status` Tauri command (Rust shell exposes it as part of the response when applicable). The `pending-anchors.json` non-empty check goes through `GET /games/{id}/pending-anchors` per active game.

### Restart flow (when guardrails clear)

1. User clicks Apply.
2. Frontend calls `daemon_stop` Tauri command.
3. Wait for daemon to fully exit (poll `daemon_status` until `state === "none"`, max 10s).
4. Frontend calls `daemon_start({ readonly: <new>, network: <new> })`.
5. `useDaemon` context updates; views re-mount with new connection.

## 7. Type-sync test extension — game state subset

Extends `tests/test_daemon_types_ts_in_sync.py` (or adds a sibling `tests/test_game_types_ts_in_sync.py`) with the **UI-consumed subset only**:

```python
GAME_STATE_FIELDS = {
    # GameConfig
    "seed", "ruleset", "max_players", "max_rounds", "board_size",
    # GameState top-level
    "config", "players", "current_round", "game_over", "winner",
    # PlayerState (UI-consumed subset)
    "name", "coins", "reputation", "upgrades",
    "vouchers_held", "active_deals", "active_treaties", "resources",
}
```

~18 fields total. Each must appear as a TS string literal OR as an interface field (regex tolerant of either form) in `app/src/types/game.ts`.

**Rationale for subset, not full schema** (Mike's amendment A): pinning every `PlayerState` field over-couples the test to schema noise (`helped_last_round`, `skip_next_move`, `apology_used`, `toasted`, `position`, `win_condition`, `vouchers_issued`, `promises`) that the UI deliberately does not consume. Schema additions to non-UI fields should not break the type-sync test.
